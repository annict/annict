## 手順

### ステップ 1: 作業計画書の作成

1. `docs/plans/template.md` をコピーして `docs/plans/1_doing/sync-YYYYMMDD-{3桁連番}.md` を作成する（YYYYMMDD は実行日）
   - 連番は `docs/plans/` 配下の全ディレクトリから `sync-YYYYMMDD-` で始まるファイルを検索し、既存の最大連番 + 1 を付与する（初回は `001`）
   - 例: `sync-20260227-001.md`（同日2回目なら `sync-20260227-002.md`）
2. タイトルを「プロジェクト間ファイル同期 YYYY/MM/DD」とする
3. 不要なセクション（「要件」「実装ガイドラインの参照」「非機能要件」「採用しなかった方針」「参考資料」「設計」）を削除する

### ステップ 2: カテゴリ別の並列差分調査

**重要**: コンテキスト制限を回避するため、カテゴリごとに Task tool（subagent_type: `Explore`）を使って並列で差分調査を行う。

各 Task tool には以下を指示する：

- 指定されたカテゴリのファイルを全プロジェクト間で比較する
- 存在しないファイルはスキップする
- 差分がないファイルは「差分なし」と明記する
- プロジェクト名のみの差異は差分として報告しない
- 実質的な内容の違い（設計方針、設定値、セクションの有無等）を報告する
- **スキップ済みの差分一覧に記載されている差分は無視する**（レポートに含めない）
- **スキップ済みの差分一覧に記載されている差分が解消されているか（全プロジェクトで同一になっているか）も併せて報告する**

### ステップ 3: スキップ済み差分のクリーンアップ

ステップ 2 の調査結果を元に、「スキップ済みの差分」に記載されている項目のうち、差分が解消されている（全プロジェクトで同一になっている）ものを特定し、該当の項目をこのファイルの「スキップ済みの差分」セクションから削除する。削除した項目があれば、作業計画書のタスクリスト内に「スキップ済み差分のクリーンアップ結果」として記録する。

### ステップ 4: 差分レポートをタスクリストに記入

作業計画書のタスクリストセクションに、以下の形式で記入する。フェーズ 1 は固定の初期タスク、フェーズ 2 以降に各差分の詳細と採用方針チェックボックスを記載する：

```markdown
## タスクリスト

### フェーズ 1: 採用方針の決定

- [ ] **1-1**: 以下の差分レポートを確認し、各差分の採用方針を記入する
  - 各差分の「採用方針」チェックボックスにチェックを入れる
  - 必要に応じて「その他」に補足を記入する
  - **重要**: 「スキップ（対応不要）」が選ばれた差分は `.claude/commands/sync.md` の「スキップ済みの差分」セクションに追記するタスクも含めること
  - 「スキップ（あとで対応）」が選ばれた差分は別途作業計画書を作成するタスクを追加すること
  - 他プロジェクトへの反映は、各プロジェクトで `/sync` を実行して対応する
  - **担当**: ユーザー

### フェーズ 2: 差分レポート

#### 差分 1: [差分の概要]

**対象ファイル**: `go/docs/validation-guide.md`

**差分の内容**:

- **Annict**: [Annict での記述内容の要約]
- **Mewst**: [Mewst での記述内容の要約]
- **Wikino**: [Wikino での記述内容の要約]

**採用方針**:

- [ ] Annict の方式を採用
- [ ] Mewst の方式を採用
- [ ] Wikino の方式を採用
- [ ] スキップ（対応不要）
- [ ] スキップ（あとで対応）
- [ ] その他: <!-- 記入 -->

**推奨理由**: [なぜその方式を推奨するかの根拠を記載。推奨なしの場合は「推奨なし（どちらでも良い）」と記載]

---
```

### ステップ 5: ユーザーへの報告

以下の内容をユーザーに報告する：

1. 作成した作業計画書のパス
2. 発見した差分の件数（カテゴリ別の内訳）
3. 「各差分の **採用方針** のチェックボックスにチェックを入れてください。記入後にタスクリストを更新します。」というメッセージ

### ステップ 6: スキップ済み差分の更新

ユーザーが採用方針で「スキップ（対応不要）」を選んだ差分について、このファイルの「スキップ済みの差分」セクションに概要を追記する。

```markdown
- `go/CLAUDE.md`: Go バージョンの違い（プロジェクト固有のため）
```

ユーザーが採用方針で「スキップ（あとで対応）」を選んだ差分について、別途作業計画書を作成するタスクをタスクリストに追加する。

## 採用方針の推奨ガイダンス

差分レポートを作成する際、各差分に対して以下の推奨を付記する。ユーザーが判断しやすくなるよう、**推奨（Recommended）** タグを適切な選択肢に追加する。

### 推奨の判断基準

**重要**: プロジェクト数の多数決で推奨を決めない。内容の正確さ・新しさで判断する。

| 状況                               | 推奨                         |
| ---------------------------------- | ---------------------------- |
| 一方がより正確・最新の内容を含む   | その方式を推奨               |
| 一方がより限定的・安全な設定である | その方式を推奨               |
| プロジェクト固有の事情がある       | スキップを推奨               |
| どちらでも良い（好みの問題）       | 推奨なし（ユーザーに委ねる） |

### 推奨の付け方

差分レポートの採用方針チェックボックスに、推奨する選択肢に `(Recommended)` を付記する。さらに、**なぜその方式を推奨するかの理由**を採用方針の直後に記載し、ユーザーが根拠を把握した上で判断できるようにする。

```markdown
**採用方針**:

- [ ] Annict の方式を採用
- [ ] Mewst/Wikino の方式を採用 (Recommended)
- [ ] スキップ（対応不要）
- [ ] スキップ（あとで対応）
- [ ] その他: <!-- 記入 -->

**推奨理由**: Mewst/Wikino の方式はより最新のバージョンに対応しており、正確な設定値を使用しているため。
```

### 推奨の例

- **バージョンの違い**: 最新バージョンを使用しているプロジェクトを推奨
- **glob パターンの精度**: より限定的・正確なパターンを推奨
- **ドキュメントの充実度**: より詳細・正確に記載しているプロジェクトを推奨
- **数値の不一致**: 実際のファイル数と合っている方を推奨
- **CI 設定の記載**: 実際のファイル構成と合っている方を推奨
- **新しい概念の追加**: 追加したプロジェクトを推奨（他プロジェクトに未反映なだけ）

## 注意事項

- 存在しないファイルはスキップする
- 差分がないファイルは「差分なし」と明記する
- プロジェクト名のみの差異は差分として報告しない
- 差分調査は Task tool で並列処理してコンテキスト制限を回避する
- 3 プロジェクト間で完全一致が確認されたファイル（`go/.air.toml`、`rails/.standard.yml` 等）は比較対象外
