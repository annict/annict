# コードレビュー <比較対象ブランチ名> <参考ドキュメントのパス>

現在のブランチを比較対象ブランチと比較してコードレビューを行ってください。

## 引数

`$ARGUMENTS` は以下の形式で渡されます：

- 形式: `<比較対象ブランチ名> <参考ドキュメントのパス>`
- 比較対象ブランチ名: 省略時は `main`
- 参考ドキュメントのパス: 省略可。指定時は参考ドキュメントの内容を踏まえてレビューを行う
  - 作業計画書（Markdown ファイル）: 実装すべき仕様・要件との整合性をチェックする
  - PR 情報ドキュメント（`/review-pr` スキルから渡される Markdown ファイル）: PR 本文・Changelog・CVE 情報・修正ループ結果などを踏まえてレビューする
  - その他任意の Markdown ドキュメント: 内容に応じてレビューの参考情報として活用する

例:

- `/review main docs/private/plans/1_doing/feature.md` → main と比較、作業計画書を参考ドキュメントとして参照
- `/review main tmp/review-pr/pr-123.md` → main と比較、PR 情報ドキュメントを参考ドキュメントとして参照（`/review-pr` からの呼び出しで使われる形式）
- `/review feature-branch` → feature-branch と比較
- `/review` → main と比較

## 手順

1. **引数の解析**: `$ARGUMENTS` から比較対象ブランチ名と参考ドキュメントのパスを取得（ブランチ名省略時は main、参考ドキュメントは省略可）
   - `$ARGUMENTS` の最初のスペース区切りトークンを比較対象ブランチ名として使用する
   - `git branch -a` で指定されたブランチが存在するか確認する。存在しない場合はエラーを報告して終了する
   - 解析結果を明示する: 「比較対象ブランチ: `{ブランチ名}`、現在のブランチ: `{現在のブランチ名}`」
2. **差分の取得**: `git diff <比較対象ブランチ>...HEAD --name-only` で変更ファイル一覧を取得
3. **参考ドキュメントの読み込み**（指定時のみ）: 参考ドキュメントを読み込み、レビューに活かすべき情報（実装すべき仕様・要件、PR 本文・Changelog・CVE 情報、修正ループ結果など）を把握する
4. **過去のレビューの確認**: `docs/private/reviews/` と `docs/private/reviews/done/` から同一ブランチのレビューファイル（`{現在のブランチ名}-*.md`）を検索する
   - 存在する場合はすべて読み込み、「ファイルごとのレビュー結果」「設計改善の提案」セクションから **対応方針にチェック（`[x]`）が入っている問題** を抽出する
   - これらを「対応方針決定済みの問題」として把握し、ステップ 8 のレビュー時に参照する
   - 対応方針が「現状維持」「後で対応」等の場合、その問題は解消されずにコード上に残っている可能性がある
5. **レビュードキュメントの作成**: `docs/private/reviews/template.md` をコピーして `docs/private/reviews/{現在のブランチ名}-{3桁連番}.md` を作成
   - 連番は `docs/private/reviews/` と `docs/private/reviews/done/` 配下の全ディレクトリから同一ブランチ名のファイルを検索し、既存の最大連番 + 1 を付与する
   - 例: `sign-up-1-1-001.md`（同一ブランチの 2 回目のレビューなら `sign-up-1-1-002.md`）
6. **レビュー情報の記入**: ブランチ名、変更ファイル数、変更行数を記入。参考ドキュメントが指定されている場合はそのパスも記入する
7. **変更ファイル一覧の記入**: 実装ファイル、テストファイル、設定ファイルに分類して記入
8. **ファイルごとのレビュー**: 各ファイルに対して以下を実行
   - ファイルを読み込む
   - テンプレートの「参照するガイドライン」セクションに記載されているガイドラインを確認
   - ファイルの種類に応じた該当ガイドラインに従っているかチェック
   - 参考ドキュメントが指定されている場合、その内容を踏まえてチェック（作業計画書なら設計との整合性、PR 情報ドキュメントなら Changelog・CVE 情報・修正ループ結果の妥当性など）
   - **過去のレビューで対応方針が決定済みの問題は再指摘しない**: 同じ問題を検出した場合でも、過去のレビューで開発者が対応方針を選択済みの問題は新たな指摘として記載しない。ただし、過去の判断の前提が変わっている場合（例: 「後で対応」とした機能が既に実装済みなのに未対応）は再指摘してよい
   - 問題点があれば「ファイルごとのレビュー結果」に記録（ガイドラインへの参照を含める）
   - **問題がないファイルは「変更ファイル一覧」のチェックボックスにチェックを入れるだけで OK**（「ファイルごとのレビュー結果」に記載しない）
9. **参考ドキュメントとの整合性チェック**（参考ドキュメント指定時のみ）: 参考ドキュメントの内容に基づいて以下を確認する
   - 作業計画書の場合: 記載された要件・仕様がすべて実装されているか。未実装や乖離があれば問題点として記録する
   - PR 情報ドキュメントの場合: PR 本文に記載された Changelog・CVE 情報・修正ループ結果と実装の整合性を確認する
   - その他の参考ドキュメントの場合: 記載内容と実装の整合性を確認する
10. **総合評価の記入**: Approve / Request Changes / Comment を判定し、総評を記入
11. **完了報告**: レビュードキュメントのパスと、主要な問題点のサマリーを報告

## レビューの観点

各ファイルをレビューする際は、以下の優先順位で注意してください：

1. **参考ドキュメントとの整合性**（参考ドキュメント指定時）: 参考ドキュメントの内容（作業計画書の仕様・要件、PR 情報ドキュメントの Changelog・CVE 情報など）と実装に乖離がないか
2. **セキュリティ**: CSRF 対策、XSS 対策、SQL インジェクション対策が適切か
3. **アーキテクチャ**: 依存関係のルールに従っているか
4. **既存コードとの一貫性**: 類似の処理が既存コードにある場合、そのパターンに従っているか
5. **命名規則**: ファイル名、関数名、変数名がガイドラインに従っているか
6. **エラーハンドリング**: エラーが適切に処理されているか
7. **テスト**: 実装に対応するテストが追加されているか

## 出力形式

レビュー完了後、以下の形式で報告してください：

```
## レビュー完了

**ドキュメント**: docs/private/reviews/{ファイル名}.md
**参考ドキュメント**: {パス}（指定時のみ表示）

**評価**: [Approve / Request Changes / Comment]

**サマリー**:
- 問題点の数: N件
- 必須対応: N件
- 推奨対応: N件
- 参考ドキュメントとの乖離: N件（参考ドキュメント指定時のみ表示）

**主要な問題点**:
1. [問題点1の概要]
2. [問題点2の概要]
...
```
