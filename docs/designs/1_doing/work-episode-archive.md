## 概要

<!--
ガイドライン:
- この機能が「何を」実現するのかを簡潔に説明
- ユーザーにとっての価値や背景を記述

-->

作品（Work）とエピソード（Episode）にアーカイブ機能を導入し、データをイミュータブルに保ちながら作品とエピソードの統合・分離を可能にする。

アーカイブされたデータは利用者に表示されるが、新規の記録やステータス変更は不可となる。これにより、作品を別の作品のエピソードに統合したり、エピソードを独立した作品に分離する運用が可能になる。

**目的**:

- Annict DB でのデータ整理を効率化する（作品の統合、エピソードの分離）
- 既存データをイミュータブルに保ち、移行コストを最小化する
- 利用者の既存の記録やステータスを維持しながら、データ構造を改善できる

**背景**:

- 現状、エピソードとして登録したものを単独の作品にしたり、逆に作品をエピソード化する運用ニーズがある
- 従来の animes テーブル導入案は約25テーブルの `work_id` / `episode_id` を更新する必要があり、変更が大きい
- アーカイブ機能を導入することで、既存データを変更せずに統合・分離を実現できる

### 設計方針: イミュータブルデータ

本設計では「イミュータブルデータ」の方針を採用する。既存データの `work_id` / `episode_id` などの参照を更新せず、古いデータをアーカイブして新しいデータを作成する。

**この設計を採用した理由**:

1. **移行コストが低い**: 約25テーブルの参照を更新する必要がない
2. **データを壊すリスクが低い**: 既存データは変更されないため、問題が発生しにくい
3. **ロールバックが容易**: `status` を `published` に戻せば元に戻せる
4. **利用者の記録を維持**: 既存の `work_records`, `episode_records`, `library_entries` などはそのまま残る

**トレードオフ**:

- アーカイブされたデータが蓄積される
- 「実質的に同じ作品・エピソード」が複数存在する状態になる
- 利用者が必要に応じて記録し直す必要がある

### データベース設計

#### status enum 型の作成

```sql
-- 作品・エピソードの状態を表す enum 型を作成
CREATE TYPE public.work_status AS ENUM ('published', 'archived', 'deleted');
CREATE TYPE public.episode_status AS ENUM ('published', 'archived', 'deleted');
```

#### works テーブルへの変更

```sql
-- works テーブルに status と archive_message カラムを追加
ALTER TABLE public.works ADD COLUMN status public.work_status NOT NULL DEFAULT 'published';
ALTER TABLE public.works ADD COLUMN archive_message VARCHAR;

-- インデックス（公開中の作品の検索用）
CREATE INDEX index_works_on_status ON public.works(status) WHERE status = 'published';

-- 既存の unpublished_at カラムからデータを移行
UPDATE public.works SET status = 'archived' WHERE unpublished_at IS NOT NULL;
UPDATE public.works SET status = 'deleted' WHERE deleted_at IS NOT NULL;

-- 移行後、unpublished_at カラムは将来的に削除
```

#### episodes テーブルへの変更

```sql
-- episodes テーブルに status と archive_message カラムを追加
ALTER TABLE public.episodes ADD COLUMN status public.episode_status NOT NULL DEFAULT 'published';
ALTER TABLE public.episodes ADD COLUMN archive_message VARCHAR;

-- インデックス（公開中のエピソードの検索用）
CREATE INDEX index_episodes_on_status ON public.episodes(status) WHERE status = 'published';

-- 既存の unpublished_at カラムからデータを移行
UPDATE public.episodes SET status = 'archived' WHERE unpublished_at IS NOT NULL;
UPDATE public.episodes SET status = 'deleted' WHERE deleted_at IS NOT NULL;

-- 移行後、unpublished_at カラムは将来的に削除
```

### 状態の分類

作品・エピソードには以下の状態が存在する：

| 状態 | `status` の値 | 操作 | 用途 |
|-----|--------------|------|------|
| 公開 | `published` | 可能 | 通常の状態 |
| アーカイブ | `archived` | 不可 | 統合・分離後の旧データ、編集者が誤って登録したデータ |
| 削除 | `deleted` | 不可 | 削除されたデータ |

#### 表示ルール

| 表示場所 | 公開 | アーカイブ | 削除 |
|---------|------|----------|------|
| 全体の一覧（検索結果、シーズン別、人気作品など） | 表示 | 非表示 | 非表示 |
| ユーザー関連の一覧（視聴一覧、記録一覧など） | 表示 | 表示 | 非表示 |
| 詳細ページ（直接URLアクセス） | 表示 | 表示（案内メッセージ付き） | 非表示 |

**既存カラムとの関係**:

- `unpublished_at`: `status` に統合。既存データは `status = 'archived'` に移行後、カラムを将来的に削除
- `deleted_at`: `status` に統合。既存データは `status = 'deleted'` に移行後、カラムを将来的に削除。削除イベントの記録は別テーブルで管理する

### 作品を別の作品のエピソードに統合する操作

例: 作品A「エヴァンゲリオン:DEATH (TRUE)²」を作品B「新世紀エヴァンゲリオン」のエピソードに統合する

```
1. 作品Aのエピソードをすべてアーカイブ（status = 'archived'）
2. 作品Bに新しいエピソードを作成（作品Aの情報を元に）
3. 作品Aをアーカイブ（status = 'archived', archive_message に「新世紀エヴァンゲリオンに統合されました」などを設定）
```

**結果**:

- 作品Aに紐付く `work_records`, `library_entries` などはそのまま残る
- 作品Aにアクセスした利用者には「アーカイブされています。新世紀エヴァンゲリオンに統合されました」と表示
- 利用者は必要に応じて作品Bに記録し直せる

### エピソードを独立した作品に分離する操作

例: 作品A「涼宮ハルヒの憂鬱」のエピソード「笹の葉ラプソディ」を作品Bに分離する

```
1. 作品Bを新規作成
2. 作品Bにエピソード（本編）を作成
3. 元のエピソードをアーカイブ（status = 'archived', archive_message に「笹の葉ラプソディ（作品）に分離されました」などを設定）
```

**結果**:

- 元のエピソードに紐付く `episode_records` はそのまま残る
- 元のエピソードにアクセスした利用者には「アーカイブされています。笹の葉ラプソディ（作品）に分離されました」と表示
- 利用者は必要に応じて新しい作品Bに記録し直せる

### エピソードなし作品へのエピソード追加

`no_episodes = true` の作品（映画など）に対しても、エピソード（本編）を1つ作成する。

これにより：

- すべての作品にエピソードが存在することが保証される
- 放送予定（slots）の登録が可能になる
- 作品への記録（レビュー）とエピソードへの記録（視聴ログ）を分けられる

**「レビュー」と「視聴ログ」の区別**:

- `work_records` = レビュー（作品全体についての包括的な感想）
- `episode_records` = 視聴ログ（各話についての感想）

映画の場合、作品ページでレビューを書き、エピソード（本編）ページで視聴ログを書くという使い分けになる。

### アーカイブされた作品・エピソードの表示

#### 作品ページ

アーカイブされた作品にアクセスした場合：

```
┌─────────────────────────────────────────────┐
│ ⚠️ この作品はアーカイブされています          │
│                                             │
│ [archive_message の内容]                    │
│ 例: 「新世紀エヴァンゲリオンに統合されました」 │
│                                             │
│ この作品への新しい記録やステータス変更は      │
│ できません。                                 │
└─────────────────────────────────────────────┘

（以下、通常の作品ページの内容）
```

#### 操作の制限

アーカイブされた作品・エピソードでは以下の操作を禁止する：

- ステータス変更（見たい、見てる、見た、など）
- 記録の追加（work_records, episode_records）
- コレクションへの追加

既存の記録やステータスは表示される（閲覧のみ可能）。

## ボツ案: animes テーブル導入

### 概要

`animes` テーブルを導入し、作品（Work）とエピソード（Episode）を統一的に識別する案。`animes.parent_id` が NULL なら作品、NULL でなければエピソードとして扱う。

詳細は [animes テーブル導入 設計書](../4_wont_do/animes-table.md) を参照。

### メリット

- 変換後も同じ `anime_id` を維持できる（URL が変わらない）
- 既存の記録（`work_records`, `episode_records` など）が自動的に引き継がれる
- 長期的にはデータがクリーンに保たれる（1つの作品 = 1つの anime）

### 採用しなかった理由

1. **移行コストが高い**
   - 約25テーブルに `anime_id` カラムを追加し、既存の `work_id` / `episode_id` から値を移行する必要がある
   - 移行中のデータ整合性の維持が難しい

2. **データ構造の複雑化**
   - `animes` テーブルが増えることで、データ構造の理解が難しくなる
   - `works` / `episodes` テーブルとの関係（`animes has_one works`, `animes has_one episodes`）が直感的でない

3. **API 利用者にとって使いにくい**
   - Web API で `anime` リソースを取得した後、`parent_id` を見て作品かエピソードかを判別する必要がある
   - 現在の `Work` / `Episode` という明確な分離の方が API 利用者にとって理解しやすい

4. **変換の頻度が低い**
   - 作品↔エピソードの変換は日常的な操作ではなく、データ整理時のみ発生
   - 頻度の低い操作のために大規模な移行を行うのは割に合わない

### 比較

| 観点 | animes テーブル案 | アーカイブ機能案（採用） |
|------|------------------|------------------------|
| 移行コスト | 高い（約25テーブル） | 低い（2テーブルに status enum + archive_message） |
| データ構造の理解しやすさ | 複雑（3テーブルの関係） | シンプル（既存構造を維持） |
| API の使いやすさ | 判別ロジックが必要 | Work/Episode が明確 |
| 変換後の URL | 維持される | 変わる |
| 既存記録の引き継ぎ | 自動 | 手動（利用者が記録し直す） |
| ロールバック | 困難 | 容易 |
