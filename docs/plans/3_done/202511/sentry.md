## 概要

<!--
ガイドライン:

Go版AnnictにSentry（エラー追跡サービス）を導入し、本番環境でのエラー監視・追跡を可能にします。
Rails版と同様にSentryを使用することで、エラーの集約、スタックトレースの可視化、ユーザーコンテキストの付与が実現できます。

**目的**:

- 本番環境でのエラーをリアルタイムに検知・追跡する
- エラー発生時のコンテキスト（ユーザーID、リクエスト情報）を記録する
- パニック発生時のスタックトレースを自動キャプチャする

**背景**:

- 現在Go版ではログ出力のみでエラー追跡機能がない
- Rails版では既にSentryを使用しており、運用ノウハウがある
- 本番環境でのエラー可視性を高め、迅速な問題解決を可能にする

### 機能要件

<!--
「ユーザーは〇〇できる」「システムは〇〇する」という形式で記述
箇条書きで簡潔に
-->

- システムは未処理のエラーを自動的にSentryに送信する
- システムはパニック発生時にスタックトレースをSentryに送信する
- システムはエラー発生時に現在のユーザーID（ログイン中の場合）をコンテキストに付与する
- システムはHTTPリクエスト情報（URL、メソッド、ヘッダー）をエラーに付与する
- 開発者はSentry Dashboardでエラーの集約・分析ができる

- **セキュリティ**: パスワード、トークンなどのセンシティブデータはSentryに送信しない
- **パフォーマンス**: Sentry送信はバックグラウンドで行い、レスポンス遅延を最小化する
- **可用性**: Sentryサービスが利用不可でもアプリケーションは正常動作する
- **トレースサンプリング**: 本番環境ではサンプルレート0.5（50%）でトレースを収集

### 技術スタック

- **Sentry Go SDK**: `github.com/getsentry/sentry-go` (v0.39.0)
- **HTTP統合**: `github.com/getsentry/sentry-go/http` (sentryhttp)

### アーキテクチャ

```
┌─────────────────────────────────────────────────────────┐
│ HTTPリクエスト                                           │
└─────────────────────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────────────────────┐
│ sentryhttp ミドルウェア                                  │
│ - リクエスト情報をキャプチャ                              │
│ - パニックリカバリー + Sentry送信                         │
│ - Hubをコンテキストに注入                                │
└─────────────────────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────────────────────┐
│ 認証ミドルウェア                                         │
│ - ユーザーコンテキストをSentryに設定                      │
└─────────────────────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────────────────────┐
│ Handler / UseCase                                       │
│ - sentry.CaptureException() でエラー送信                 │
└─────────────────────────────────────────────────────────┘
```

### コード設計

#### パッケージ構成

```
internal/
├── sentry/
│   └── sentry.go          # Sentry初期化・ヘルパー関数
├── middleware/
│   └── sentry.go          # Sentryミドルウェア（ユーザーコンテキスト設定）
```

#### 主要な関数・構造体

**internal/sentry/sentry.go**:
```go
// Config はSentryの設定を保持する
type Config struct {
    DSN              string
    Environment      string
    TracesSampleRate float64
    Debug            bool
}

// Init はSentryを初期化する
func Init(cfg Config) error

// CaptureError はエラーをSentryに送信する
func CaptureError(ctx context.Context, err error)
```

**internal/middleware/sentry.go**:
```go
// SentryUserContext はユーザーコンテキストをSentryに設定するミドルウェア
func SentryUserContext(next http.Handler) http.Handler
```

#### 環境変数

| 環境変数 | 説明 | 必須 | デフォルト |
|---------|------|------|----------|
| `ANNICT_SENTRY_DSN` | Sentry DSN | Yes (本番) | - |
| `ANNICT_SENTRY_ENVIRONMENT` | 環境名 | No | `APP_ENV` の値 |
| `ANNICT_SENTRY_TRACES_SAMPLE_RATE` | トレースサンプリングレート | No | `0.5` |
| `ANNICT_SENTRY_DEBUG` | デバッグモード | No | `false` |

### セキュリティ設計

**センシティブデータのフィルタリング**:

`BeforeSend` フックを使用して、以下のデータをマスクまたは除外する：

- リクエストヘッダー: `Authorization`, `Cookie`, `X-CSRF-Token`
- リクエストボディ: `password`, `token`, `secret` を含むフィールド
- クエリパラメータ: `token`, `key` を含むパラメータ

### テスト戦略

- **単体テスト**: `internal/sentry/sentry.go` の初期化ロジック
- **統合テスト**: ミドルウェアでのユーザーコンテキスト設定
- **手動テスト**: Sentry Dashboardでのイベント受信確認
