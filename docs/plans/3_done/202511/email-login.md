## 概要

メールアドレスとパスワードなしでログインできる「パスワードレスログイン」機能を実装します。ユーザーはメールアドレスを入力し、届いた 6 桁の数字コードを入力するだけでログインできます。

Rails 版では「ログイン用 URL をメールで送信 → URL をクリック」という方式でしたが、Go 版では「6 桁の数字コードをメールで送信 → 数字を入力」という方式に変更します。これにより、モバイルアプリでの対応が容易になり、将来的な拡張性が向上します。

**目的**:

- パスワードを覚える必要がなく、ユーザーの利便性を向上
- セキュリティリスクの低減（パスワードの漏洩リスクがない）
- モバイルアプリでの対応を容易にする（コード入力フォームのみ）

**背景**:

- Rails 版にはメールアドレスでログインする機能があったが、URL クリック方式だった
- Go 版では 6 桁コード方式に変更し、モバイルアプリ対応を見据えた設計にする
- パスワードログインとメールログインを統合した UI を提供し、UX を向上させる

### 機能要件

#### ログインフロー

1. **メールアドレス入力**:
   - ユーザーは `/sign_in` にアクセスし、メールアドレスのみを入力する
   - システムはメールアドレスが登録済みかを確認する
2. **ログイン方法の自動判定**:
   - システムは `users.encrypted_password` の有無を確認する：
     - パスワードが存在する場合 → パスワードログイン画面へリダイレクト（既存機能）
     - パスワードが存在しない場合 → 6 桁コード送信 → 6 桁コード入力画面へリダイレクト（新機能）
3. **6 桁コードの送信**（パスワードが存在しない場合）:
   - 6 桁の数字コードがメールで送信される
   - コードの有効期限は 15 分
4. **6 桁コードの入力**:
   - ユーザーは届いた 6 桁コードを入力する
   - システムはコードを検証し、正しければセッションを作成してログイン完了
5. **エラーハンドリング**:
   - メールアドレスが登録されていない場合、エラーメッセージを表示
   - 6 桁コードが間違っている場合、エラーメッセージを表示
   - 6 桁コードの有効期限が切れている場合、再送信を促すメッセージを表示

#### セキュリティ要件

- 6 桁コードは暗号学的に安全な乱数生成器で生成する
- 6 桁コードは bcrypt でハッシュ化してデータベースに保存する
- コードの試行回数を制限する（5 回まで）
- Rate Limiting を適用する（1 分間に 3 回まで）
- コードの有効期限は 15 分
- 使用済みコードは無効化する

#### セキュリティ

- **暗号化**: 6 桁コードは bcrypt でハッシュ化してデータベースに保存
- **試行回数制限**: 6 桁コードの入力は 5 回まで（超過時はコードを無効化）
- **Rate Limiting**: メール送信は 1 分間に 3 回まで（Redis で管理）
- **有効期限**: 6 桁コードは 15 分で無効化
- **使用済みコードの無効化**: コードは 1 回のみ使用可能

#### ユーザビリティ（UX）

- **シンプルな UI**: メールアドレス入力のみのフォーム
- **明確なフィードバック**: エラーメッセージと成功メッセージを適切に表示
- **再送信機能**: コードの有効期限が切れた場合、再送信ボタンを表示
- **国際化対応**: すべてのメッセージを日本語と英語で提供

#### パフォーマンス

- **メール送信**: Resend API を使用し、バックグラウンドジョブで非同期送信（riverqueue）
- **データベースクエリ**: インデックスを活用し、高速なクエリを実現

### データベース設計

新しいテーブル `email_login_codes` を作成します：

```sql
CREATE TABLE email_login_codes (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  code_digest VARCHAR(255) NOT NULL,  -- bcryptでハッシュ化された6桁コード
  attempts INT NOT NULL DEFAULT 0,     -- 試行回数
  expires_at TIMESTAMP WITH TIME ZONE NOT NULL,  -- 有効期限
  used_at TIMESTAMP WITH TIME ZONE,    -- 使用済み時刻
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- インデックス
CREATE INDEX idx_email_login_codes_user_id ON email_login_codes(user_id);
CREATE INDEX idx_email_login_codes_expires_at ON email_login_codes(expires_at);
CREATE INDEX idx_email_login_codes_used_at ON email_login_codes(used_at);
```

#### テーブル設計の方針

**設計判断**: ログイン専用のテーブル `email_login_codes` を作成（汎用テーブルではない）

**理由**:

1. **YAGNI 原則**: 将来の機能（メールアドレス変更など）は必要になったときに実装する
2. **責務の分離**: ログインとメール変更は目的が異なり、要件も異なる
   - ログイン: `user_id` → コード確認 → セッション作成
   - メール変更（将来）: `user_id` + 新メールアドレス → コード確認 → メール更新
3. **セキュリティ**: 別々のテーブルで、コードの混同リスクを回避
4. **保守性**: 各機能が独立しているため、変更の影響範囲が限定的
5. **障害対応**: 1 つの機能の障害が他の機能に影響しない

**将来の拡張時の方針**:

- メールアドレス変更機能を実装する際は、新しいテーブル `email_change_codes` を作成
- コード生成・検証などの共通ロジックは、ユースケース層で抽象化・共通化
- テーブルは別々に管理、コードは DRY 原則に従って共通化

### API 設計（エンドポイント）

#### 1. メールアドレス入力

- **URL**: `GET /sign_in`
- **説明**: メールアドレス入力フォームを表示
- **ハンドラー**: `internal/handler/sign_in/show.go`
- **レスポンス**: HTML（メールアドレス入力フォーム）

#### 2. メールアドレス送信と自動判定

- **URL**: `POST /sign_in`
- **説明**: メールアドレスを受け取り、`users.encrypted_password` の有無で自動的に分岐
- **ハンドラー**: `internal/handler/sign_in/create.go`
- **リクエストボディ**:
  ```json
  {
    "email": "user@example.com"
  }
  ```
- **レスポンス**:
  - パスワードが存在する場合: `302 Found` → `/sign_in/password`（パスワードログイン画面）
  - パスワードが存在しない場合: 6 桁コード送信 → `302 Found` → `/sign_in/code`（6 桁コード入力画面）
  - セッションにメールアドレスとユーザー ID を保存

#### 3. 6 桁コード入力画面

- **URL**: `GET /sign_in/code`
- **説明**: 6 桁コード入力フォームを表示
- **ハンドラー**: `internal/handler/sign_in_code/show.go`
- **レスポンス**: HTML（6 桁コード入力フォーム）

#### 4. 6 桁コード検証

- **URL**: `POST /sign_in/code`
- **説明**: 6 桁コードを検証し、セッションを作成
- **ハンドラー**: `internal/handler/sign_in_code/create.go`
- **リクエストボディ**:
  ```json
  {
    "code": "123456"
  }
  ```
- **レスポンス**: `302 Found` → `/`（ログイン成功）

#### 5. 6 桁コード再送信

- **URL**: `PATCH /sign_in/code`
- **説明**: 6 桁コードを再送信（リソースの更新として扱う）
- **ハンドラー**: `internal/handler/sign_in_code/update.go`
- **レスポンス**: `302 Found` → `/sign_in/code`（フラッシュメッセージ: コードを再送信しました）

### セキュリティ設計

#### ユーザー列挙攻撃への対応方針

**設計判断**: メールアドレスが登録されていない場合、明示的にエラーメッセージを表示する

**理由**:

1. **UX 優先**: Google、GitHub、Twitter など多くの大手サービスが同様の挙動を採用している
2. **ユーザビリティ**: 曖昧なエラーメッセージはユーザーの混乱とサポートコストの増加を招く
3. **実効性**: パスワードリセットやアカウント作成フォームでも同様の情報が得られるため、隠蔽の効果は限定的
4. **対策の存在**: Rate Limiting（1 分間に 3 回まで）で大量試行を防止、将来的に Cloudflare Turnstile でボット対策を追加予定

**リスク認識**: ユーザー列挙攻撃（大量のメールアドレスを試して登録済みユーザーのリストを作成）の可能性は認識しているが、上記の理由から許容範囲内と判断

#### 6 桁コードの生成

- `crypto/rand` を使用して暗号学的に安全な乱数を生成
- 6 桁の数字（000000 〜 999999）
- bcrypt でハッシュ化してデータベースに保存

#### Rate Limiting

Redis を使用して以下の制限を実装：

- **メール送信**: `1 分間に 3 回まで`（キー: `email_login:send:{email}`, TTL: 60 秒）
- **コード検証**: `1 分間に 5 回まで`（キー: `email_login:verify:{email}`, TTL: 60 秒）

**開発環境での無効化**:

- 環境変数 `ANNICT_DISABLE_RATE_LIMIT=true` を設定することで、開発環境では Rate Limiting を無効化できる
- パスワードリセット機能と同様の実装（既存の `ANNICT_DISABLE_RATE_LIMIT` を再利用）
- 本番環境では必ず有効にする（デフォルト: 有効）

#### 試行回数制限

- データベースの `attempts` カラムで試行回数を管理
- 5 回失敗したらコードを無効化（`used_at` に現在時刻を設定）
- エラーメッセージ: 「試行回数が上限に達しました。新しいコードを送信してください。」

#### 有効期限

- 6 桁コードは 15 分で無効化
- `expires_at` カラムで管理
- 有効期限切れのコードは自動的に無効化

#### 使用済みコードの無効化

- コードは 1 回のみ使用可能
- 使用後は `used_at` カラムに現在時刻を設定

### 実装方針

#### 既存システムとの関係

- **既存のハンドラー**: `internal/handler/sign_in/` は既にパスワードログイン用として存在

- セッション管理は既存の実装を再利用

**重要**: タスク 2-1 の前に、既存の `internal/handler/sign_in/` を `internal/handler/sign_in_password/` にリファクタリングする準備タスクが必要
