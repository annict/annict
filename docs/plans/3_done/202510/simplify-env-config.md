# 環境変数設定のシンプル化 設計書

## 概要

現在のGoプロジェクトでは、`ANNICT_ENV`環境変数を使って`.env.{environment}`ファイルを切り替えていますが、これは一般的なGoプロジェクトのアプローチではありません。
開発環境とテスト環境で異なる設定ファイルを読み込む必要性が薄いため、Goの標準的なアプローチに合わせてシンプル化します。

**目的**:

- Goコミュニティで一般的な環境変数管理方法に統一する
- 開発時とテスト時で同じ設定を使うことでメンテナンスコストを削減
- シークレット管理ツール経由で環境変数を設定するパターン（`.env`ファイル不要）との併用をサポート

**背景**:

- 現在は`ANNICT_ENV`で環境ごとにファイルを切り替えているが、Rails/Node.jsスタイルでGoコミュニティでは少数派
- テスト用の`.env.test`ファイルが存在せず、テストが失敗している
- 開発環境でもテスト環境でも基本的に同じ設定を使用するため、ファイルを分ける必要性が薄い
- シークレット管理ツールを使う場合、環境変数はメモリ上に存在する

## 要件

### 機能要件

- `.env`ファイル1つのみを使用する（`.env.{environment}`の仕組みを廃止）
- 事前に環境変数が設定されている場合は、その値を優先して使用する
- テスト実行時も開発サーバ起動時も同じ設定ファイル（`.env`）を参照する
- テスト用のデータベースは別のPostgreSQLサーバー（docker-compose.ymlで定義）を使用する
- `ANNICT_ENV`環境変数は開発/本番の判定にのみ使用する（デフォルト: `development`）

### 非機能要件

**セキュリティ**:

- 秘密情報（APIキーなど）を含む`.env`ファイルは引き続き`.gitignore`で管理
- 事前に設定された環境変数は`.env`ファイルより優先される

**保守性**:

- 環境変数ファイルの数を減らし、メンテナンスコストを削減
- 既存の実装との互換性を保ちながら段階的に移行

## 設計

### 環境変数読み込みの優先順位

変更後の優先順位（上から順に優先）：

1. **事前に設定された環境変数**（メモリ上に存在、シークレット管理ツールやホスティング環境が提供）
2. **`.env`**（環境変数設定ファイル、gitignore対象）

### ファイル構成の変更

**変更前**:

```
.env.example            # サンプル設定（バージョン管理対象）
.env                    # （複数の.env.{environment}ファイルを読み込む想定、現在は存在しない）
.env.development        # （想定されているが存在しない）
.env.development.local  # （想定されているが存在しない）
.env.test               # （想定されているが存在しない）
```

**変更後**:

```
.env.example  # サンプル設定（バージョン管理対象）
.env          # 実際の環境変数設定（gitignore対象、開発・テスト両方で使用）
```

### データベース設定

開発環境とテスト環境で同じPostgreSQLサーバーを使用し、データベース名のみを切り替えます：

**開発環境**:

- 接続先: `postgres://postgres:password@postgresql:5432/annict_development?sslmode=disable`
- 環境変数: `DATABASE_URL`（`.env`ファイルで設定）

**テスト環境**:

- 接続先: `postgres://postgres:password@postgresql:5432/annict_test?sslmode=disable`
- 環境変数: `DATABASE_URL`（コード内で`ANNICT_ENV=test`の時に動的に切り替え）

`.env`ファイルでは開発環境用のDATABASE_URLを設定し、テスト実行時は`ANNICT_ENV=test`の環境変数でテスト用DBを使うように分岐します。

### コード設計

**internal/config/config.go**:

- 環境別ファイル読み込みのロジックを削除
- `.env` ファイルのみを読み込み（シンプル化）
- `getRequiredEnv()`ヘルパー関数を削除し、`os.Getenv()`を直接使用（過度な抽象化を避ける）
- 既存の環境変数が設定されている場合（シークレット管理ツールやホスティング環境経由）はそちらを優先
- `ANNICT_ENV`は`development`/`production`/`test`の判定にのみ使用

### テスト戦略

- `config_test.go`を更新して、`.env`ファイル読み込みのテストを追加
- 事前に設定された環境変数が優先されることをテスト
- 既存のテストは維持しつつ、不要なテストケースは削除

### 実装方針

- `.env.example`の内容を参考に開発者が`.env`を作成する方式（従来通り）
- `config.Load()`関数を簡略化（`.env`のみ読み込み）
- `CLAUDE.md`の環境変数設定のセクションを更新

### 実装しない機能（スコープ外）

以下の機能は今回の実装では**実装しません**：

- **本番環境の設定**: 本番環境では`.env`ファイルを使わず、環境変数を直接設定する想定（将来的にKubernetesのSecretやAWS Secrets Managerなどを使用）
- **環境別の詳細な設定管理**: 環境ごとに大きく異なる設定が必要になった場合は、その時点で再検討する
