## 設計改善の提案

設計改善の提案はありません。

Worker を薄い Adapter に変更し、ビジネスロジックを UseCase に移動する設計は、作業計画書の目標アーキテクチャに沿っており、以下の点で適切です：

- **インターフェース分離**: `ExpiredTokenCleaner`、`ExpiredSignInCodeCleaner` インターフェースにより、Worker と UseCase が疎結合に
- **依存方向の遵守**: Worker（Presentation 層）→ UseCase（Application 層）→ Repository（Domain/Infrastructure 層）
- **命名規則の遵守**: UseCase のファイル名（`cleanup_expired_tokens.go`）、構造体名（`CleanupExpiredTokensUsecase`）が `{action}_{entity}` 規則に準拠
- **ログ出力**: `slog.InfoContext`/`slog.ErrorContext` を正しく使用
- **エラーハンドリング**: `fmt.Errorf` で適切にラップ
- **テスト**: Worker テストが UseCase 経由に更新され、正常系・異常系・境界値をカバー
