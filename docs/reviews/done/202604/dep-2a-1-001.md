### 計画との相違点（設計改善）

作業計画書の設計例では `InsertSendEmail(ctx, args worker.SendEmailArgs)` という汎用メソッド 1 つを定義していたが、実装では以下の 3 つの専用メソッドに分割されている：

- `InsertSignInCodeEmail(ctx, email, code, locale)`
- `InsertSignUpCodeEmail(ctx, email, code, locale)`
- `InsertPasswordResetEmail(ctx, email, resetURL, locale)`

**評価**: これは設計の改善として妥当。理由:

1. UseCase が `worker.Build*Email()` を呼ぶ必要がなくなり、UseCase → worker の依存が完全に除去される
2. Dispatcher の API がドメイン固有の操作として明確になる（引数が `SendEmailArgs` ではなくプリミティブ値）
3. メール構築ロジックのカプセル化が向上する

計画が想定していた「UseCase → worker の依存禁止」を、計画以上に徹底した設計になっている。
