### `go/internal/handler/sign_in_password/create.go`

**ステータス**: 要確認

**チェックしたガイドライン**:

- [@go/docs/handler-guide.md](/workspace/go/docs/handler-guide.md) - HTTP ハンドラーガイドライン
- [@go/docs/architecture-guide.md#Handlerの処理フロー変更](/workspace/go/docs/architecture-guide.md) - Handler は薄い Adapter

**問題点・改善提案**:

- **[作業計画書#Handlerの処理フロー変更]**: `flashJSON` の構築が UseCase 呼び出しの前に行われている

  リファクタリング前は flashJSON 構築はパスワード検証成功後（認証成功が確定した後）に行われていましたが、現在は UseCase 呼び出し前に移動しています。認証失敗時にも flashJSON のエンコード処理が実行されます。

  flash メッセージは「ログイン成功」を表す Presentation 層の関心事です。作業計画書の目標フローでは Handler は「HTTP リクエストのパース → UseCase 呼び出し → 結果に応じたレスポンス」のみを担当する薄い Adapter とされています。flash メッセージの構築を UseCase 内部に移すことで、Handler をより薄くでき、かつ認証失敗時の無駄な処理も排除できます。

  ```go
  // 現在のコード（handler/create.go）
  flashMsg := session.Flash{
      Type:    session.FlashSuccess,
      Message: i18n.T(ctx, "sign_in_success"),
  }
  flashJSON, err := flashMsg.ToJSON()
  // ...
  result, err := h.authenticateByPasswordUC.Execute(ctx, usecase.AuthenticateByPasswordInput{...}, flashJSON)
  ```

  **修正案**:

  flash メッセージの構築を `AuthenticateByPasswordUsecase.Execute` 内部（セッション作成の直前）に移す。

  ```go
  // usecase 内部で flash メッセージを構築
  flashMsg := session.Flash{
      Type:    session.FlashSuccess,
      Message: i18n.T(ctx, "sign_in_success"),
  }
  flashJSON, err := flashMsg.ToJSON()
  if err != nil {
      slog.ErrorContext(ctx, "flashメッセージのエンコードエラー", "error", err)
      flashJSON = ""
  }
  sessionResult, err := uc.createSessionUC.Execute(ctx, nil, user.ID, user.EncryptedPassword, flashJSON)
  ```

  ただし、これは `CreateSessionUsecase` の既存インターフェースとの整合性の問題もあるため、現状維持も合理的です。

  **対応方針**:

  - [ ] 現状のまま（`CreateSessionUsecase` のインターフェース変更は別タスクで対応）
  - [x] その他（下の回答欄に記入）
