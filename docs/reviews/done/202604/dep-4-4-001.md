### `go/internal/usecase/get_supporter_status.go`

**ステータス**: 要修正

**チェックしたガイドライン**:

- [@go/docs/architecture-guide.md](/workspace/go/docs/architecture-guide.md) - データの流れ、レイヤー間の依存関係
- [@go/CLAUDE.md#ビューモデル・ユースケース](/workspace/go/CLAUDE.md) - UseCase の命名規則

**問題点・改善提案**:

- **[@go/docs/architecture-guide.md#データの流れ]**: UseCase の Output が `repository.*` 型を参照している

  `GetSupporterStatusOutput` が `*repository.StripeSubscriber` と `*repository.GumroadSubscriber` を直接使用しています。しかし、既存の UseCase（`GetPopularWorksUsecase` → `[]model.WorkWithDetails`、`GetUserCalendarUsecase` → `*model.UserCalendar`）はすべて `model.*` 型を使用しており、Handler が `repository` パッケージを import する必要がないパターンになっています。

  現在の実装では Handler（`show.go`）が `repository` パッケージを import しており、フェーズ 6 の depguard ルール（Handler → Repository 禁止）が有効化された時点でビルドが壊れます。

  ```go
  // 現在のコード（34-39行目）
  type GetSupporterStatusOutput struct {
      IsStripeActive    bool
      IsGumroadActive   bool
      StripeSubscriber  *repository.StripeSubscriber
      GumroadSubscriber *repository.GumroadSubscriber
  }
  ```

  **修正案**:

  `model` パッケージに `StripeSubscriber` と `GumroadSubscriber` のモデル型を定義し、Repository で query 型から model 型への変換を行い、UseCase Output では model 型を使用する。

  ```go
  // internal/model/stripe_subscriber.go（新規）
  type StripeSubscriber struct {
      ID                     int64
      StripeCustomerID       string
      StripeStatus           string
      StripeCurrentPeriodEnd time.Time
      StripeCancelAt         sql.NullTime
      // ... 必要なフィールド
  }

  // internal/usecase/get_supporter_status.go（修正）
  type GetSupporterStatusOutput struct {
      IsStripeActive    bool
      IsGumroadActive   bool
      StripeSubscriber  *model.StripeSubscriber
      GumroadSubscriber *model.GumroadSubscriber
  }
  ```

  これにより Handler は `repository` パッケージを import する必要がなくなり、他の UseCase と一貫したパターンになります。
