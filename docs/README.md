# docs ディレクトリガイド

このディレクトリでは「サービス仕様書」「システム仕様書」「作業計画書」「レビュー」など、Annict のドキュメントを管理している。

## ディレクトリ構成

```
docs/
├── README.md           # このファイル
├── specs/              # サービス仕様書 (ユーザーが直接体験する機能)
│   ├── template.md     # 仕様書テンプレート
│   └── {モデル名}/     # モデルごとのディレクトリ
│       ├── overview.md # モデルの定義・状態遷移 (必要な場合)
│       └── xxx.md      # 各機能の仕様書
├── system/             # システム仕様書 (ユーザーが直接体験しないシステム内部の仕組み)
│   ├── template.md     # 仕様書テンプレート
│   └── {機能名}/       # 機能ごとのディレクトリ
│       └── overview.md # 機能の仕様書
└── private/            # 公開しない作業ドキュメント (作業計画書、レビュー、ヘルプ草稿など)
    ├── README.md       # private/ 配下の補足
    ├── plans/          # 作業計画書
    ├── reviews/        # レビュー
    ├── helps/          # ヘルプページ草稿
    └── demo/           # デモ用ページ
```

## 各ドキュメントの役割

| ドキュメント                  | 記述する内容                                                   | ライフサイクル                     |
| ----------------------------- | -------------------------------------------------------------- | ---------------------------------- |
| サービス仕様書 (`specs/`)     | ユーザーが直接体験する機能の現在の状態と、なぜそうなっているか | 長期 (継続的に更新)                |
| システム仕様書 (`system/`)    | ユーザーが直接体験しないシステム内部の仕組みの現在の状態       | 長期 (継続的に更新)                |
| 作業計画書 (`private/plans/`) | 何をどう変えるか (変更の差分)                                  | 短期 (タスク完了後にアーカイブ)    |
| レビュー (`private/reviews/`) | 実装に対するセルフレビューの結果                               | 短期 (対応完了後に `done/` へ移動) |
| Git 履歴                      | 過去の状態                                                     | 永続                               |

仕様書と作業計画書・レビューはライフサイクルが異なるため、独立したディレクトリで管理している。

## 仕様書の分類: `specs/` か `system/` か

新しい仕様書を作成するときは、以下の基準で配置先を決める。

**判断基準**: 「ユーザーがその機能の存在を直接体験するか?」

- **Yes** → `docs/specs/` (サービス仕様)
  - 例: ページ作成、Wikiリンク、サジェスト、サイドバー、エラーページ (404 画面)
- **No** → `docs/system/` (システム仕様)
  - 例: フィーチャーフラグ機構、日時表示の内部実装、Markdown フォーマッタ

ユーザーが「機能」として認識しないシステム内部の基盤 (ミドルウェア、フォーマッタ、内部ライブラリなど) は `system/` に配置する。ハイブリッドな機能 (ユーザー視点と実装視点の両方を持つもの) は `specs/` 側に集約し、1 ファイル内で「## 仕様」(ユーザー視点) と「## 設計」(実装視点) のセクションで内部分離する。

## サービス仕様書 (`specs/`)

- 「ユーザーが直接体験する機能の現在の状態」と「なぜそうなっているか」を記述する
- 実装が完了したら、仕様書を最新の状態に更新する
- 過去の状態はGit履歴で参照する
- テンプレート: [specs/template.md](specs/template.md)

### モデルごとのディレクトリ構成

仕様書はモデル (ドメインの名詞) ごとにディレクトリを分け、その中に機能 (動詞) の仕様書を配置する。

```
docs/specs/
├── template.md
├── user/
│   ├── overview.md           # Userモデルの定義・状態遷移・他モデルとの関係
│   ├── sign-up.md
│   ├── sign-in.md
│   └── password-reset.md
├── space/
│   ├── overview.md
│   ├── create.md
│   └── ...
└── page/
    ├── overview.md
    ├── create.md
    ├── edit.md
    └── ...
```

**ファイルの種類**:

| ファイル            | 記述する内容                             | 視点               |
| ------------------- | ---------------------------------------- | ------------------ |
| `overview.md`       | モデルの定義・状態遷移・他モデルとの関係 | 「これは何か」     |
| 機能の仕様書 (動詞) | 操作の振る舞い・バリデーション・権限     | 「どう操作するか」 |

- `overview.md` はモデルの静的な性質を記述する場所であり、操作に紐づく仕様 (バリデーションルール、権限など) は各機能の仕様書に書く
- `overview.md` は必要なモデルにのみ作成する (状態遷移や他モデルとの関係がないシンプルなモデルでは不要)

**分類ルール**:

- 操作対象となるモデルのディレクトリに仕様書を配置する
  - 例: 「ユーザーがページを作成する」→ 操作対象はページなので `page/create.md`
- モデルに分類しにくい横断的な機能 (検索、通知など) は、その機能自体を名詞として独立したディレクトリにする
  - 例: `search/`、`notification/`
- 仕様書が1つしかない場合でも、一貫性のためにディレクトリを作成する

## システム仕様書 (`system/`)

- 「ユーザーが直接体験しないシステム内部の仕組みの現在の状態」と「なぜそうなっているか」を記述する
- ミドルウェア、フォーマッタ、内部ライブラリなど、ユーザーが「機能」として認識しない基盤を対象にする
- 実装が完了したら、仕様書を最新の状態に更新する
- テンプレート: [system/template.md](system/template.md)

### ディレクトリ構成

機能ごとにディレクトリを作成し、`overview.md` にその機能の仕様を集約する。

```
docs/system/
├── template.md
├── feature-flag/
│   └── overview.md
├── datetime/
│   └── overview.md
└── formatter/
    └── overview.md
```

## 作業計画書 (`private/plans/`)

- 「何をどう変えるか」という変更内容を記述する
- 対応する仕様書へのリンクを「仕様書」セクションに記載する
- タスク完了後に対応する仕様書を新しい状態に更新する
- テンプレート: [private/plans/template.md](private/plans/template.md)

## レビュー (`private/reviews/`)

- 実装完了後のセルフレビュー結果を記録する
- ファイル名は `{ブランチ名}-{3桁連番}.md` の形式 (例: `sign-up-1-1-001.md`)
- レビュー対応が完了したら `done/` ディレクトリに移動する
- テンプレート: [private/reviews/template.md](private/reviews/template.md)

## 運用フロー

### 新しい機能を作る場合

1. `private/plans/` に機能の概要・要件・設計・採用しなかった方針・タスクリストを記述する
2. タスクを実装する
3. 実装完了後、`specs/` または `system/` に仕様書を作成する (設計判断や採用しなかった方針も含める)
4. 作業計画書をアーカイブする

### 既存機能を変更する場合

1. 作業計画書を作成し、対応する仕様書にリンクする
2. 変更内容を作業計画書に記述する
3. タスクを実装する
4. タスク完了後、仕様書を新しい状態に更新する
5. 作業計画書をアーカイブする

## 採用しなかった方針

### 仕様書にドラフトステータスを設ける

未実装機能の仕様書を `specs/` に `Status: draft` として配置し、実装完了後にステータスを削除する方針を検討した。

**不採用の理由**:

- `specs/` の性質が曖昧になる。仕様書を開いたとき、それが現在のシステムの状態を反映しているかどうかをステータスで判断する必要が生じる
- 既存機能の変更時には作業計画書に「変更内容」を書く必要があり、結局 `plans/` にも仕様的な記述が必要になる。新機能は `specs/` にドラフト、既存変更は `plans/` に記述、と場所が分かれて統一感がない
- `specs/` を「現在のシステムの状態」だけに限定するほうが、信頼できる情報源として機能する

### サービス仕様とシステム仕様を厳密に分離する

すべての機能について、サービス視点の記述を `docs/specs/` に、実装視点の記述を `docs/system/` に 2 ファイルへ厳密分離する方針を検討した。

**不採用の理由**:

- 同じ機能を 2 ファイルに分けると 1 機能の全体像が見えなくなり、常に 2 ファイル間の同期コストが発生する
- ハイブリッド機能 (ページ編集など) は 1 ファイル内の `## 仕様` / `## 設計` セクションで十分に読み分けられる
- 個人開発で読み手が限定的な現状において、厳密分離のコストは便益に見合わない
